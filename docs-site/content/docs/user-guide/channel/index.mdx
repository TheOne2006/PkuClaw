Channel 是 PkuClaw 和外部平台之间的连接层。它负责：

1. 接收用户消息、菜单点击、卡片回调等输入；
2. 把 realtime 回复、后台 loop 通知、文件或图片发送到具体平台；
3. 把平台里的用户、会话、群聊标识转换成 PkuClaw 能理解的 target。

## 配置分工

Channel 配置通常分两层：

| 配置层 | 文件 | 作用 |
| --- | --- | --- |
| 启动配置 | `configs/config.toml` | 平台 app id、secret 环境变量、事件模式、API base。 |
| Runtime 通知目标 | `configs/runtime/runtime.json` | 后台 loop 默认把通知发到哪个 channel 和 target。 |

例如飞书启动配置在：

```toml
[feishu]
app_id = "cli_xxx"
app_secret_env = "FEISHU_APP_SECRET"
event_mode = "websocket"
```

后台通知目标在：

```json
{
  "notifications": {
    "policy": "important_only",
    "default_channel": "feishu",
    "default_target_type": "open_id",
    "default_target_id": "ou_xxx"
  }
}
```

## Realtime 和 loop 的差别

| 场景 | target 从哪里来 |
| --- | --- |
| 用户主动发消息 | channel 事件里带着当前会话，PkuClaw 优先回到这次触发的会话。 |
| 用户点 Quick Action | channel 事件里带着操作者或会话，PkuClaw 回到触发入口。 |
| 后台 loop 主动通知 | 没有天然会话，必须看 `runtime.json` 里的默认通知目标或 loop 覆盖目标。 |

因此，能收到 realtime 回复，不代表后台 loop 一定能主动通知。后台通知需要单独配置 target。

## 安全边界

不要把这些内容写进可提交文件：

- app secret；
- access token；
- open id；
- chat id；
- cookie；
- 私有 webhook 或回调地址。

可提交文件里只放占位符和示例。
